# パイプライン手動運用・動作確認マニュアル

このドキュメントは、「浦安市給食・献立ナビ」のバックエンド（手動パイプライン実行）に関する、安全な操作手順と動作確認の観点を忘れないようにまとめたものです。

---

## 運用前提知識
- マスター実行スクリプトは `src/run_pipeline.py` です。
- **引数なし** (`python src/run_pipeline.py`) で実行すると、**システム日時から自動的に「翌月」のデータをターゲット**にして全自動処理（ダウンロード〜解析〜結合〜Firebaseアップロード）が走ります。
- したがって、**当月末頃に「来月分の献立が公開されたかな？」というタイミングで月1回実行するだけ**で運用が完了する設計になっています。

---

## 安全な動作確認テスト手順
運用前にロジックの修正を行った場合などは、以下の順序でテストを行うことで、不意の実データ破壊やエラーを防ぐことができます。

### Step 1: まだ公開されていない「未来の月」での空振りテスト（ドライラン近似）
PDFが市役所のサイトに存在しない場合の挙動確認です。
```bash
# （例）まだ公開されていない半年後の年月を指定
python src/run_pipeline.py --year 2026 --month 12
```
**【確認ポイント】**
- ダウンロードスクリプト (`src/download_menu.py`) がPDFのリンクを発見できず、以下のようなログを残して `Exit code 1` で終了すること。
- 「Artifacts have been successfully generated」などの誤った完了メッセージが出ないこと。
- Firebaseへ空のJSONがアップロードされないこと。

### Step 2: 既存の月（最新データ）でのフルテストと差分チェック
過去あるいは当月のデータを指定し、実際にすべての処理（Gemini文字起こし含む）を通すテストです。
これを行うことで、システム（AI解析ロジック）が壊れていないかを確認します。

1. **既存データのバックアップ**
   思わぬデータ破壊を防ぐため、実行前に既存データのバックアップを取ります。
   ```bash
   cp data/2026_3.json data/2026_3.json.bak
   ```

2. **パイプラインの特定月実行**
   アップロード処理も含まれるため、確実に動く月を指定します。
   ```bash
   python src/run_pipeline.py --year 2026 --month 3
   ```

3. **生成結果の差分比較**
   処理完了後、バックアップしたJSONと今回新しく生成されたJSONの中身を比較します。
   - **お箸フラグ (needs_chopsticks)**: OpenCVの画像解析がズレていないか（以前の判定と同じか）。
   - **メニュー内容 (menu_items)**: Geminiの文字起こしが著しく欠落したり、行のパースに失敗していないか。
   - ※AIの特性上、「豚汁」が「とんじる」になる等の微妙な表記揺れは発生しますが許容範囲とします。

4. **Firebase反映確認**
   - Webアプリ側（`https://kondate-navi.web.app`）をリロードして、該当月のデータが正しく表示される（またはコンソールの `Network` タブで新しいデータの取得に成功している）ことを確認します。
   - キャッシュ回避のため、アップロード時には `cache-control: no-store` が付与されています。

---

> [!NOTE]  
> 今後のフェーズで GitHub Actions 化を行う場合も、この「存在しない月のエラー監視」と「異常時の処理中断」が担保されているため、安全にcronを回すことができます。
