# システム詳細設計書 (Kondate Navi Pipeline)

このドキュメントでは、浦安市学校給食の献立PDFからデータを抽出し、Webアプリへと配信するバックエンド処理（Pythonパイプライン）の内部実装詳細について解説します。
各Pythonスクリプトがどのように呼び出され、引数や月またぎの処理をどう捌いているか、そしてAI抽出・画像解析の正確なロジックをリバースエンジニアリングの観点で記載しています。

---

## 1. 全体アーキテクチャと処理フロー (`src/run_pipeline.py`)

本システムのマスターオーケストレーターです。他のPythonファイル群を `subprocess.run` で順番に独立したプロセスとして呼び出します。途中のプロセスが異常終了（0以外のExit Code）を返した場合、パイプライン全体が直ちに安全停止します。

### 月の自動判定と引数 (`--year`, `--month`)
- **引数なしの場合**: 実行時のシステム時刻 (`datetime.now()`) を基準に、**自動的に「翌月」のデータをターゲット**とします（例：11月に実行されたら12月を、12月に実行されたら翌年1月を計算してセットします）。
- **引数ありの場合**: `--year 2026 --month 3` のように明示的に指定された月をターゲットにします（過去の再実行やテスト用）。

### 呼び出しシーケンス
1. **[Step 1] PDFのダウンロード**
   `subprocess.run(["python", "src/download_menu.py", year, month])`
2. **[Step 2] AI文字起こしと画像解析**
   `subprocess.run(["python", "src/transcribe_pdf.py", "--force", "--year", year, "--month", month])`
   ※マスターから呼ばれた場合は `--force` が付与され、過去のJSONが存在しても強制再描画します。
3. **[Step 3] データの集計・結合**
   `subprocess.run(["python", "src/generate_data.py", "--year", year, "--month", month])`
4. **[Step 4] Firebaseへのアップロード**
   `subprocess.run(["python", "src/upload_to_firebase.py", "data/YYYY_M.json"])`

## 2. モジュール詳細解説

### 2.1 PDF自動取得スクリプト (`src/download_menu.py`)
- **呼び出し引数**: `python src/download_menu.py [year] [month]` (引数なしの場合は当月)
- **目的**: 浦安市の公式ページ（`index.html`）をスクレイピングし、対象年月の「小学校給食（第一調理場・第二調理場）」の献立PDFをダウンロードします。
- **実装詳細**:
  - `BeautifulSoup` を使用して指定URLをパースします。
  - 「令和X年Y月分献立予定表」という和暦変換された見出しブロックを探し、その直下にある「第一調理場」「第ニ調理場」のPDFリンク (`<a>` タグ) を探し出します。表記揺れ（漢数字の二とカタカナのニなど）を吸収する正規表現を使用しています。
  - 取得したPDFは `downloads/YYYYMM/` ディレクトリに保存されます。
  - **エラー検知**: 対象月のリンクが1つも見つからなかった場合（市役所がまだ公開していない場合）、`sys.exit(1)` を呼び出して異常終了します。これにより、上位のパイプラインも空打ちを検知して安全に停止します。

### 2.2 画像解析・AI文字起こし (`src/transcribe_pdf.py`)
- **目的**: PDFを画像化し、Gemini（文字起こし）とOpenCV（お箸マーク検知）の2系統でデータを抽出します。
- **AI文字起こし (Gemini 3.1 Pro)**:
  - PDFの1ページ目を `PyMuPDF` で300DPI相当のPNG画像としてレンダリングします（透過背景対策済み）。
  - 画像と「プロンプト」を `Vertex AI` の Gemini エンドポイントに送信します。プロンプトには厳格なJSONスキーマと、「ひらがなは漢字に直す」「表の外の情報は無視する」「献立名は勝手に正規化せず原型をとどめる」といった強力な抽出制約が書かれています。
- **お箸フラグの画像解析判定 (OpenCV)**:
  AIによる「手書き丸印の認識」は不確実なため、ピクセルベースの幾何学的な画像処理で `needs_chopsticks` を判定します。
  1. **動的アンカーの発見**: まずファイル内のテキスト（Words）から「日」というヘッダーのX座標を探し出します。
  2. **真円の抽出**: `cv2.findContours` で輪郭を抽出し、真円度（面積と半径の比率）が `0.5 < x < 1.2` であり、かつ大きさが適切（半径20〜50px）な「丸印（〇）」だけをリストアップします。この時、「日」列のX座標周辺にある〇だけを対象にします。
  3. **日付とのマッチング**: カレンダー上の数字（1〜31）の座標と、発見された丸印の中心座標の直線距離を計算します。距離が60px以内であれば重なっていると判定し、その日付の `needs_chopsticks` を `True` とします。

### 2.3 データ集計・ファイル整形 (`src/generate_data.py`)
- **呼び出し引数**: `--year YYYY --month M` 
- **目的**: `transcribe_pdf.py` が吐き出した複数の施設別JSONデータを、1つの構造化ファイル (`data/YYYY_M.json`) に結合整理します。
- **処理対象**: 指定された年月の `results/YYYYMM/*.json` だけを読みに行きます。
- **実装詳細**:
  - 読み込んだJSONのメタ情報 `meta.facility_name` (第一調理場・第二調理場など) を、日次データ（`daily_menus`）の配列内に直接コピー注入します。
  - 同様に、Webアプリ内でのフロントエンド処理（ReactのKeyやソート）を容易にするためのID (`date_id`: "2026-3-2") や `year`, `month` といった一意キーを追加します。
  - これにより、施設ごとにネストされていたデータ構造が、「日付と施設ごとに完全に独立したフラットな配列構造」へと最適化されます。

### 2.4 Firebaseへのデプロイ (`src/upload_to_firebase.py`)
- **呼び出し引数**: アップロードしたいJSONファイルのパス（例：`data/2026_3.json`）
- **目的**: 集計済みのローカルJSONデータを Firebase Storage 上へ直接書き込みます。
- **実装詳細**:
  - `firebase-admin` SDKとデフォルト認証情報(ADC)を使用。
  - **キャッシュ対策 (`no-store`)**: 
    アップロード時にHTTPヘッダーとして `Cache-Control: no-store, max-age=0` メタデータを強制付与します。本Webアプリのキャッシュ戦略上、Firebase Storageからデータをフェッチする際、ブラウザにしつこく残る古いJSONキャッシュを回避して常に最新を読み取らせるための最重要実装です。
